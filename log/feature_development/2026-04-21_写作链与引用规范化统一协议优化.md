# 2026-04-21 写作链与引用规范化统一协议优化

## 背景

在 OCR、NER、citation graph、academic note 与 workflow metadata 主线逐步收口后，写作链仍存在几处明显断点：

1. `citation_normalizer.py` 返回的仍是旧式解析碎片，无法与 `citation_network_analyzer.py` / `citation_formats.py` 共享同一 citation record。
2. `paper_polisher.py` 仍是直接绑定单一 LLM 的旧模块，没有纳入统一任务层，也没有多后端可发现能力。
3. `reverse_outline_analyzer.py` 仍是独立分析器，没有进入统一执行协议，workflow 无法稳定记录其 backend/provider/model。
4. `stage5_write.py` / `stage6_polish.py` 还没有把草稿快照、review queue、quality flags、execution summary 正式写回 `ResearchProject.stage_metadata`。

这些问题会直接影响：

- 写作链与前面分析链之间的数据连续性
- workflow 恢复、审计、前端展示与人工复核
- 多后端能力在写作链上的可切换性

## 本轮目标

1. 先把新增设计约束写回锚点文档 `MODULE_OPTIMIZATION_DESIGN_2026-04-21.md`
2. 重构 `citation_normalizer.py`，建立统一 citation record schema
3. 重构 `paper_polisher.py` 与 `reverse_outline_analyzer.py`，让它们成为统一任务层门面
4. 补齐 `reverse_outline` 在 `UnifiedTaskExecutor` / `TaskManager` / `module_adapters` 中的正式接入
5. 更新 Stage 5 / Stage 6，把草稿与审校元数据写回 workflow
6. 补正式测试并完成回归验证

## 具体改动

### 1. 锚点文档补充

在 `docs/project/MODULE_OPTIMIZATION_DESIGN_2026-04-21.md` 中新增了本轮增量约束，明确：

- `citation_normalizer.py` 要升级到统一 citation record schema
- `paper_polisher.py` 要转为统一任务层门面
- `reverse_outline_analyzer.py` 要转为统一写作审校协议门面
- `stage5_write.py` / `stage6_polish.py` 需要写回 `stage_metadata / review_queue / quality_flags`

### 2. CitationNormalizer 重构

重写 `modules/citation_normalizer.py`，当前已稳定输出统一 citation record，至少包含：

- `raw_text`
- `normalized_citation`
- `type`
- `title`
- `authors`
- `author`
- `year`
- `journal_or_publisher`
- `journal`
- `publisher`
- `pages`
- `volume`
- `issue`
- `doi`
- `url`
- `confidence`
- `needs_review`
- `backend`
- `provider`
- `model`

同时保留了：

- `detect_format`
- `parse_citation`
- `normalize`
- `normalize_batch`
- `convert_format`
- `create_citation_normalizer`

这样 Stage 7 与 formatter/analyzer 就可以共享同一引用数据基础。

### 3. PaperPolisher 重构

重写 `modules/paper_polisher.py`，它现在是统一任务层上的稳定门面：

- 通过 `TaskManager.paper_polish(...)` 执行真正的润色任务
- 保留 `polish_paragraph` 与 `process_document` 旧接口
- 保留 Word track changes 兼容逻辑
- 统一输出：
  - `polished_text`
  - `revision_notes`
  - `backend`
  - `provider`
  - `model`
  - `confidence`
  - `needs_review`

### 4. ReverseOutlineAnalyzer 重构

重写 `modules/reverse_outline_analyzer.py`，让它具备两层能力：

1. 统一任务层调用
   - 通过 `TaskManager.reverse_outline(...)` 请求统一能力
2. 本地启发式 fallback
   - 在任务层失败或强制脚本模式时，仍能稳定返回 outline review

统一输出字段包括：

- `section_word_counts`
- `section_ratios`
- `logical_gaps`
- `deviation_flags`
- `suggestions`
- `confidence`
- `needs_review`
- `backend`
- `provider`
- `model`

并保留了旧式辅助接口：

- `extract_outline`
- `detect_imbalance`
- `check_logic_gaps`
- `suggest_revisions`

### 5. 统一任务层接入 reverse_outline

更新：

- `modules/unified_task_executor.py`
- `modules/module_adapters.py`
- `modules/task_manager.py`

具体包括：

- 正式新增 `ReverseOutlineHandler`
- 在 task alias 中加入 `reverse_outline` / `outline_review`
- 新增 `ReverseOutlineAdapter`
- `TaskManager` 正式支持：
  - `get_task_options("reverse_outline")`
  - `execute_task("reverse_outline", ...)`
  - `reverse_outline(...)`
- 顺手补了 style transfer API 结果解析，避免 Stage 6 再手写清洗逻辑

### 6. Stage 5 / Stage 6 workflow 收口

#### Stage5Write

重写 `tools/workflow/stages/stage5_write.py`，现在除了写入 `paper_draft` 之外，还会把以下内容写回 metadata：

- 草稿字数
- 段落数
- 章节数
- 引用占位计数
- 是否存在参考文献章节
- literature 数量
- warning 列表

并在必要时写入：

- `review_queue`
- `quality_flags`

#### Stage6Polish

重写 `tools/workflow/stages/stage6_polish.py`，现在会统一消费：

- `paper_polish`
- `style_transfer`
- `reverse_outline`

并把以下内容写回 `stage_metadata["stage6"]`：

- `capability_snapshot`
- `requested_execution`
- `execution_summary`
- `warnings`

同时会在 outline gap / deviation flag / polish fallback 等情形下写入：

- `review_queue`
- `quality_flags`

另外，这次顺手修掉了原有 style transfer 路径里 `markdown_len` 未定义造成的旧 bug。

分别验证：
