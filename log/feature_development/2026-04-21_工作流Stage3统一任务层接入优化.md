# 2026-04-21 工作流 Stage3 统一任务层接入优化

## 背景

在完成统一任务执行层和 OCR/NER 主链协议化之后，`tools/workflow/stages/stage3_extract.py` 仍然存在两个明显短板：

- 阶段内部直接硬编码 NER/LLM 选择逻辑，没有通过统一任务层发现能力
- 阶段运行后没有把 backend/provider/model、低置信度结果、阶段执行摘要写回项目状态

这会导致工作流恢复、前端能力展示、日志审计和人工复核都缺少稳定落点。

## 本轮目标

1. 先把新的 workflow 方向写回 `MODULE_OPTIMIZATION_DESIGN_2026-04-21.md`
2. 扩展 `ResearchProject`，使其能承载阶段元数据、复核队列和 artifact 索引
3. 重构 `Stage3Extract`，改为通过 `TaskManager` 调度 NER，并回写能力快照和执行摘要
4. 增加正式测试，验证 Stage 3 与统一任务层的接入行为

## 具体改动

### 1. 锚点文档补充

在 `docs/project/MODULE_OPTIMIZATION_DESIGN_2026-04-21.md` 中新增了：

- “工作流阶段仍缺少可追溯的能力快照与复核承载结构”
- `research_project.py` 的新增约束
- `stage3_extract.py` 的新增约束
- Phase 3 新的交付标准：阶段需持久化能力快照、执行摘要和人工复核队列

### 2. ResearchProject 升级

重写 `tools/workflow/research_project.py`，保留既有公共数据类，同时补充：

- `artifacts`
- `quality_flags`
- `review_queue`
- `stage_metadata`

并补齐以下能力：

- 更完整的 `save/load`
- `set_stage_metadata`
- `get_stage_metadata`
- `add_artifact`
- `add_quality_flag`
- `add_review_item`

### 3. Stage3Extract 重构

重写 `tools/workflow/stages/stage3_extract.py`，核心变化如下：

- 通过 `TaskManager` 获取 `ner` 能力快照
- 通过统一任务层执行 NER，而不是在 stage 内硬编码某个单独模块
- 支持阶段级执行参数：
  - `ner_backend`
  - `ner_provider`
  - `ner_model`
  - `fallback_backends`
  - `categories`
  - `test_mode`
- 将以下信息写回 `project.stage_metadata["stage3"]`：
  - `capability_snapshot`
  - `requested_execution`
  - `execution_summary`
  - `review_count`
  - `warning_count`
- 将低置信度或 `needs_review` 的实体写入 `project.review_queue`
- 将阶段级风险写入 `project.quality_flags`

### 4. 测试补充

新增 `tests/test_stage3_workflow_integration.py`，验证：

- Stage 3 是否通过统一任务层执行
- 是否记录 capability snapshot
- 是否记录 requested backend
- 是否生成 execution summary
- 是否把低置信度实体写入 review queue
- `ResearchProject` 是否能持久化新的 workflow 元数据字段
