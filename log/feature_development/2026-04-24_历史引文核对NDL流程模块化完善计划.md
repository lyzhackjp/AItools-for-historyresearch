## 主要问题

1. `modules/historical_citation_verifier.py` 已经承担过多职责，解析、检索、下载、页码映射、OCR、对齐、报告逻辑都混在一起。
2. 失败状态过粗，目前大量不同原因都被压到 `download_failed`，包括 NDL 检索 0 条、候选误匹配、浏览器超时、受限下载失败、OCR 前置流程卡住。
3. NDL 检索排序还不够严格，曾出现《日本の選択》脚注命中无关题名样本的情况。

5. OCR 缺少更细粒度的单页超时、缓存和失败恢复。
6. 报告生成逻辑应该从 CLI 脚本中抽出，便于 Web/API 和批处理共用。
7. `resume` 与 `batch_resume` 脚本已经有效，但还不是正式模块接口，后续应收口成 runner/service。

## 模块化目标

建议新增包目录：

- `modules/historical_citation/`

建议拆分模块：

- `models.py`: `ParsedParagraph`、`ParsedFootnote`、`NDLSearchMatch`、`CitationCandidate`、结果状态枚举。
- `docx_parser.py`: 只负责 DOCX 正文、脚注、元数据、引号抽取。
- `candidate_builder.py`: 正文段落与脚注合并，生成候选引文。
- `ndl_search.py`: NDL SRU/API/浏览器检索、候选归一化、题名作者年份评分。
- `ndl_downloader.py`: 受限下载、登录、印刷下载、IIIF 回退、同源失败缓存。
- `page_mapping.py`: 双开页检测、书页到扫描页换算、`page_mapping_cache.json` 读写。
- `ocr_extractor.py`: PDF 页抽取、双开页左右拆分、NDL OCR 调用、OCR 清洗。
- `aligner.py`: 中文译文与日文 OCR 片段对齐，保留启发式与后续 LLM 对齐接口。
- `reporting.py`: Markdown 报告、摘要、失败清单、人工复核清单。
- `runner.py`: checkpoint、单条候选重试、批量限时控制、进程清理。

保留兼容门面：

- `modules/historical_citation_verifier.py`

下次改造时先让旧类继续作为 facade，对外 API 尽量不变，内部逐步迁移到新包，避免一次性大重构导致功能回退。

## 状态与错误码完善

建议把当前状态扩展为以下类型：

- `matched`: 已形成较可靠对照。
- `needs_manual_review`: 找到来源和上下文，但页距、置信度或候选质量需人工复核。
- `source_not_found`: NDL 检索明确无结果。
- `source_mismatch`: NDL 返回结果与脚注题名/作者/年份明显不匹配。
- `restricted_download_failed`: 找到 NDL 来源，但受限下载未成功。
- `download_timeout`: 单条候选下载或前置流程超时。
- `ocr_failed`: PDF 已获得，但 OCR 无文本。
- `alignment_failed`: OCR 有文本，但无法找到合理对应段落。
- `skipped_same_source_failed`: 同源文献已确认失败，后续同源候选跳过。

报告中应按这些状态分组，而不是只显示 `download_failed`。
